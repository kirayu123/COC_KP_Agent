# COC KP Agent 需求文档

## 1. 项目目的

COC KP Agent 是一个面向《克苏鲁的呼唤》（Call of Cthulhu, COC）跑团主持人（Keeper，以下简称 KP）的辅助应用。

本项目的主要目的不是替代 KP 自动主持跑团，而是作为 KP 的“副驾驶”：帮助 KP 管理人物卡、剧情模组、规则判定、跑团历史和神话资料，在玩家做出行动后快速整理当前局势，并给出下一步剧情推进建议、可触发线索、可用检定和 NPC 反应参考。

项目希望解决的核心问题是：

- KP 临场需要同时管理人物、NPC、线索、模组文本、规则和玩家行动，认知负担很高。
- 模组信息量大，跑团过程中翻找关键线索、地点、NPC 动机和分支条件效率低。
- COC 规则涉及技能检定、困难等级、奖励骰/惩罚骰、SAN、战斗、伤害、疯狂等内容，新手 KP 容易中断节奏查规则。
- 长团历史难以维护，上一场的玩家行动、已发现线索、NPC 状态和角色状态容易遗忘。
- 玩家行动不可预测，KP 需要快速生成合理、克制、不剧透的临场反馈。

因此，本项目的产品定位是：

> 一个本地优先、资料可追溯、规则计算可靠、由 KP 最终裁决的 COC 跑团辅助控制台。

## 2. 目标用户

### 2.1 核心用户

- COC 跑团主持人 / KP
- 新手 KP
- 需要主持长团或复杂模组的 KP
- 希望管理自定义模组、人物卡和跑团日志的 TRPG 玩家

### 2.2 次要用户

- 模组作者
- 跑团资料整理者
- 希望学习 COC 主持流程的玩家

## 3. 核心价值

### 3.1 对 KP 的价值

- 快速查看当前场景、玩家状态、NPC 状态和已发现线索。
- 根据玩家输入和历史记录生成下一步主持建议。
- 自动提示可能需要的技能检定、SAN 检定、线索触发条件和 NPC 反应。
- 将规则判定交给确定性规则引擎，减少临场查表和计算。
- 将模组、规则书、神话资料和跑团日志整合到同一个检索系统中。

### 3.2 对跑团体验的价值

- 减少主持过程中的停顿。
- 降低新手 KP 入门门槛。
- 提高线索管理和长团连续性。
- 让 AI 输出停留在“建议”层，避免越权改写剧情或替 KP 做最终裁决。

## 4. 产品边界

### 4.1 本项目要做

- 人物卡导入、编辑和状态管理。
- 剧情模组导入、结构化解析和检索。
- 跑团日志记录和长期历史摘要。
- 当前局势整理。
- 基于当前场景、模组内容、玩家行动和跑团历史生成 KP 建议。
- COC7 常见规则判定工具。
- 本地资料库整理和 RAG 检索。
- 引用来源可追溯的资料检索结果。

### 4.2 本项目第一阶段不做

- 不做完全自动主持。
- 不替 KP 强制推进剧情。
- 不直接内置有版权争议的商业模组正文。
- 不将第三方站点正文、规则书全文或用户私有资料作为商业产品随包分发。
- 不让大模型直接承担最终规则计算。
- 不做多人实时联机同步，除非 demo 稳定后再扩展。

## 5. Demo 范围

第一版 demo 应聚焦“KP 跑团控制台”，优先实现可演示的闭环：

1. 创建一个跑团。
2. 导入或创建调查员人物卡。
3. 导入一个自制短模组。
4. 显示当前场景、NPC、线索和玩家状态。
5. KP 输入玩家行动。
6. 系统给出下一步剧情建议、检定建议和可触发线索。
7. KP 接受、编辑或丢弃建议。
8. 系统记录跑团日志并更新当前局势。
9. 使用规则工具完成技能、SAN 或战斗相关判定。

推荐 demo 模组使用自制内容，例如《雨夜档案室》，避免版权风险。

## 6. 功能需求

### 6.1 团管理

- 创建、编辑、删除跑团。
- 记录团名、规则版本、时代背景、玩家人数、当前模组、当前进度。
- 支持查看最近一次跑团摘要。

### 6.2 人物卡管理

- 支持手动创建调查员。
- 支持 JSON / Markdown / 表格格式导入人物卡。
- 管理属性、技能、HP、MP、SAN、幸运、武器、道具、背景和状态。
- 支持角色状态变化记录。
- 支持从人物卡直接发起技能检定。

### 6.3 模组导入与解析

- 支持 Markdown、纯文本、PDF 导入。
- 自动提取章节、场景、NPC、地点、线索、检定点、结局分支。
- 支持 KP 手动修正自动解析结果。
- 保留原文引用，便于追溯建议来源。
- 支持“已发现线索”和“未发现线索”区分。

### 6.4 跑团控制台

控制台是 demo 的核心页面，应包含：

- 当前场景
- 当前时间线
- 玩家角色状态
- NPC 状态
- 已发现线索
- 未触发关键线索
- 模组原文片段
- 跑团日志
- AI 建议面板
- 检定工具面板
- 调查板：NPC、地点、线索、证物和事件之间的关系图
- 进度钟：仪式倒计时、NPC 怀疑度、警察介入、怪物接近等可视化进展

### 6.5 玩家行动输入与局势更新

- KP 可以输入玩家行动或对话摘要。
- 系统根据当前场景、历史和模组内容生成候选建议。
- KP 可以接受、编辑或忽略建议。
- 只有 KP 确认后的内容才写入正式跑团历史。

### 6.6 AI 剧情建议

AI 应输出：

- 下一步剧情推进建议。
- NPC 反应建议。
- 可触发线索。
- 可能需要的技能检定。
- 成功、失败、大成功、大失败的叙事结果建议。
- 节奏提醒，例如是否该补线索、收束场景或制造压力。
- 剧透风险提醒，例如某信息不应在当前阶段透露。

AI 输出必须区分：

- 模组事实
- 跑团已确认事实
- AI 推测
- 可选主持建议

### 6.7 规则判定

规则判定应由确定性规则引擎完成。

第一版至少支持：

- 普通技能检定
- 困难成功
- 极难成功
- 大成功
- 大失败
- 奖励骰
- 惩罚骰
- 对抗检定
- SAN 检定
- HP / SAN / MP 变化记录
- Roll Request：KP 可向指定角色发起某项检定请求，并记录结果

后续可扩展：

- 战斗轮
- 伤害计算
- 疯狂症状
- 追逐规则
- 成长检定

### 6.8 跑团日志与长期记忆

- 记录每次玩家行动、检定结果、KP 确认事件和状态变化。
- 自动生成本场摘要。
- 维护长期状态表：
  - 当前场景
  - 已发现线索
  - 未发现关键线索
  - NPC 状态
  - 玩家状态
  - 未解决问题
  - 已确认事实

### 6.9 本地资料库与 RAG

当前已整理本地资料：

- `resources/cthulhuclub/markdown/`：Cthulhu Club 非模组资料 1268 篇。
- `resources/rulebooks-md/`：用户本地 PDF 转 Markdown 2 份。
- `resources/curated/`：非模组资料分类索引。

第一版建议只将高优先级资料接入主知识库：

- 神话实体与神格：83 条
- 怪物与遭遇：51 条
- 人物与 NPC 原型：56 条
- 典籍、道具与禁忌知识：37 条
- 地点、年代与世界观：54 条
- 跑团与 KP 参考：2 条

共 283 条。

中优先级叙事语料和文章可作为单独风格库，不建议直接混入主检索结果。

### 6.10 Handout 与主持素材

- 支持创建和管理 handout，例如报纸剪报、信件、照片说明、名片、档案摘录、日历页。
- Handout 可以关联场景、NPC、地点、线索和玩家角色。
- Handout 可以标记为“已展示给玩家”或“仅 KP 可见”。
- 后续可支持从模板生成 handout，但第一版先以 Markdown/文本卡片为主。

## 7. 非功能需求

### 7.1 本地优先

Demo 阶段优先采用本地存储和本地索引，降低隐私和版权风险。

### 7.2 可追溯

所有 AI 建议应尽量附带来源：

- 模组原文位置
- 跑团日志记录
- 规则书页码
- 本地资料 Markdown 路径或来源 URL

### 7.3 可控

AI 不应自动改写正式跑团状态。所有关键更新必须由 KP 确认。

### 7.4 可靠

规则计算不依赖 AI 自由生成，应通过规则引擎实现。

### 7.5 可扩展

系统应支持后续扩展：

- 多团管理
- 多模组管理
- 自定义规则
- 多语言资料
- 语音转写
- 云端同步
- 插件化规则模块

## 9. 资源与版权边界

项目当前已经明确排除 `/coc-mod/` 与 `/coc-modules/` 路径，没有本地化 Cthulhu Club 模组正文。

资源使用原则：

- 允许用户本地导入自己拥有或有权使用的资料。
- 本地资料库仅用于个人学习、demo 验证和检索实验。
- 商业版本不应内置第三方站点正文、规则书全文或未授权模组。
- 若未来提供云端服务，用户资料应私有化存储，不跨用户共享。

## 10. 推荐技术路线

Demo 阶段：

- 前端：React / Next.js
- 桌面封装：Tauri 或 Electron，后续可选
- 数据库：SQLite
- 检索：SQLite FTS + 向量索引
- AI：OpenAI API 或可替换模型接口
- PDF 文本抽取：现有 `tools/build_resource_library.py`
- 资源分类：现有 `tools/categorize_resources.py`

核心原则：

- 规则引擎与 AI 分离。
- 跑团状态与 AI 建议分离。
- 原文资料与结构化摘要分离。
- 用户确认与自动推测分离。

## 11. 成功标准

第一版 demo 可视为成功，如果它能完成以下演示：

1. 导入一份人物卡。
2. 导入一份自制模组。
3. 自动识别场景、NPC 和线索。
4. KP 输入玩家行动。
5. 系统基于当前局势给出可解释的下一步建议。
6. 系统提示合理的技能检定或 SAN 检定。
7. 规则工具正确给出判定结果。
8. KP 确认后，系统更新日志和当前状态。
9. 下一轮建议能使用上一轮历史。
10. 建议结果能够追溯到模组、规则书或资料来源。
